## Maintainer Responsibilities

### Code Review

- Review pull requests within 72 hours (best effort)
- Ensure code quality, tests, and documentation
- Verify adherence to project style and conventions
- Test changes with real MIDI hardware when possible
- Provide constructive feedback to contributors

### Issue Triage

- Respond to new issues within 48 hours
- Label issues appropriately (bug, enhancement, good first issue, etc.)
- Ask for clarification or reproduction steps
- Close duplicates and invalid issues
- Prioritize critical bugs and security issues

### Release Management

- Coordinate release cycles (minor releases every 4-6 weeks)
- Update CHANGELOG.md with all changes
- Tag releases following SemVer
- Build and upload release binaries
- Write and publish release notes
- Announce releases in GitHub Discussions

### Community Engagement

- Welcome new contributors
- Answer questions in GitHub Discussions
- Recognize contributors in release notes
- Foster inclusive, respectful community
- Enforce Code of Conduct when necessary

### Documentation

- Keep documentation up-to-date with code changes
- Review documentation PRs
- Ensure examples and tutorials remain accurate
- Maintain inline code documentation (rustdoc)

## Becoming a Maintainer

We welcome contributors who demonstrate:

1. **Sustained Contribution**: High-quality code, docs, or community support over 6+ months
2. **Technical Understanding**: Deep knowledge of MIDIMon architecture and Rust best practices
3. **Review Participation**: Active code review and constructive feedback
4. **Community Values**: Respectful, inclusive, and helpful communication
5. **Commitment**: Willingness to dedicate ongoing time to project health

**How to Get Started**:

- Contribute code, documentation, or issue triage
- Participate in GitHub Discussions and code reviews
- Help other users with questions and troubleshooting
- Propose and discuss new features
- Build a track record of quality contributions

**Nomination Process**:

Maintainers are appointed by the project lead based on contribution history and project needs. If you're interested in becoming a maintainer, reach out to the project lead or express interest in GitHub Discussions.

## Maintainer Emeritus Policy

Maintainers who wish to step down or who become inactive (6+ months without engagement) may be granted **Emeritus** status:

- Recognized for their contributions
- Removed from active maintainer responsibilities
- Can return to active status if they resume contribution
- Retain access to project communication channels
